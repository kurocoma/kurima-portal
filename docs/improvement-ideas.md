## 最初に着手すべき Top 5（理由付き）

1. **S1 ログローテーション** — `portal-run.log` / `portal-error.log` は今の実装（`logging.FileHandler`）だと
   確実に無限成長する。毎日使うほど劣化が進む種類の問題で、工数は最小（半日）なのに効果が恒久的。
2. **S2 共有ログのPC別分離** — 共有ログは SharePoint 同期フォルダに固定ファイル名で書くため、
   複数PCの同時書き込みで OneDrive の同期競合が起こり得る。壊れ方が「静か」（競合コピーが増える・
   ログが割れる）で気づきにくく、S1 と同じファイル（`portal_app/log_paths.py`）を触るのでセットで済む。
3. **S4 稼働バージョンの可視化（+ O1 とセット）** — 「.py を直したのに反映されない」は README の
   トラブルシュートに載るほどの既知の罠。画面に commit ハッシュと起動時刻が出るだけで
   「直したのに直らない」調査が一瞬で終わる。工数小。
4. **S3 二重実行ガード** — このツールにはクリックポストの実決済・ヤマトB2の実取込がある。
   現状はサーバー側に排他がなく、別タブ・別PCから同じジョブを同時に走らせられる。
   起きたときの損害（二重決済・二重取込）が全アイデア中で最大なので、P1 の中では工数が中でも優先。
5. **U1 エラーメッセージの日本語対処ガイド化** — 失敗時に画面へ出るのは `Timeout 60000ms exceeded.`
   のような生の英語例外。非エンジニアの日次利用という前提では、ここが復旧時間（＝業務停止時間）を
   直接決める。B2ログインには既に原因分類（`system_error` 等）があり、土台は揃っている。

## カテゴリ別詳細 — 安定性（7 件）

### S1. 実行ログ・エラーログのローテーション導入

- **現状の課題（根拠）**: `portal_app/log_paths.py` の `setup_file_logging()` は
  `logging.FileHandler` で `portal-run.log` / `portal-error.log` に追記し続けるだけで、
  ローテーションがない。出力先は既定で SharePoint 同期フォルダ（`神里\くりまポータルエラーログ`）
  なので、肥大するほど OneDrive の同期も重くなる。
- **改善案**: `logging.handlers.RotatingFileHandler` に差し替え、サイズ上限と世代数を
  環境変数で調整可能にする（新設: `KURIMA_LOG_MAX_MB` 既定 5 / `KURIMA_LOG_BACKUP_COUNT` 既定 3）。
  変更箇所は `setup_file_logging()` 内のハンドラ生成 2 か所のみ。同期フォルダ上の rename が
  気になる場合に備え、世代数の既定は小さくしておく。
- **期待効果**: ログ肥大による同期遅延・ディスク圧迫・「開いたら重すぎて読めない」を恒久的に防止。
- **難易度**: 小 / **優先度**: P1

### S2. 共有ログのPC別ファイル名分離（SharePoint 同期競合対策）

- **現状の課題（根拠）**: `portal_app/log_paths.py` はファイル名を定数
  `RUN_LOG_NAME = "portal-run.log"` / `ERROR_LOG_NAME = "portal-error.log"` で固定しており、
  全PCが SharePoint 同期フォルダ内の**同じファイル**へ追記する。README（ログ出力の節）のとおり
  実行ログには「サーバー起動、ジョブ・CLI の開始/終了」が書かれるため、本番ホスト（8006 常駐）の
  稼働中に別PCで検証起動や CLI 実行をすると多重書き込みになり、OneDrive が競合コピーを作ったり
  同期エラーでログが割れたりする恐れがある。
- **改善案**: ファイル名にPC名を入れる（例: `portal-run-{COMPUTERNAME}.log`）。実装は
  `log_paths.py` で `os.environ.get("COMPUTERNAME", "pc")` をファイル名に埋め込むだけ。
  明示上書き用に `KURIMA_LOG_SUFFIX` も用意する。閲覧側は U2（/logs の共有ログ対応）で
  プレフィックス一致（`portal-run-*.log`）の一覧表示にすれば全PC分を横断で見られる。
- **期待効果**: 同期競合の構造的排除。障害調査時に「どのPCで起きたか」がファイル名で即わかる。
- **難易度**: 小 / **優先度**: P1

### S3. 同一ワークフローの二重実行ガード

- **現状の課題（根拠）**: `portal_app/services/progress_jobs.py` の `ProgressJobStore.start()` は
  無条件に worker スレッドを起動し、サーバー側の排他がない。ボタン無効化は
  ページ内 JS のみ（`yamato.html` の `button.disabled = true`）なので、**別タブ・別PC**
  （LAN共有運用）からは同じ `/yamato/run-selected/start` 等を同時に叩ける。このツールには
  クリックポストの実決済・ヤマトB2の実取込という不可逆操作があり、二重実行の損害が大きい。
- **改善案**: `ProgressJobStore.start()` に排他チェックを追加する。`self._jobs` を走査し、
  同じ `workflow` で `status in {"queued", "running"}` のジョブがあれば新規開始せず、
  既存の `job_id` を載せた HTTP 409 を各 `/start` エンドポイントから返す。フロントは 409 を
  受けたら既存ジョブの進捗表示に切り替える（U4 の再アタッチと同じ仕組みを流用）。
- **期待効果**: 二重決済・二重取込・Next Engine セッションの取り合いを構造的に防止。
- **難易度**: 中 / **優先度**: P1

### S4. 稼働バージョンの可視化（再起動忘れ検知）

- **現状の課題（根拠）**: `scripts/serve.ps1` は `--reload` なしで uvicorn を起動するため、
  `.py` の変更・`update.bat` での更新はプロセス再起動（`scripts/restart.bat`）まで反映されない。
  README のトラブルシュートに「.py を変更したのに反映されない」が載るほどの既知の罠だが、
  画面からは「いま動いているのがいつのコードか」を確認する手段がない（`/health` は
  `{"status": "ok"}` のみ）。
- **改善案**: `portal_app/main.py` の起動時に `git rev-parse --short HEAD` を 1 回実行して
  短縮ハッシュと起動時刻を保持（git 不在なら "unknown"）。`base.html` のフッターに
  「version: abc1234 / 起動: 07-07 08:30」を常時表示し、`/health` にも
  `version` / `started_at` を追加する。
- **期待効果**: 「更新したのに直らない」の原因切り分けが一目で終わる。LAN利用者も
  ホストの再起動漏れに気づける。O1（update.bat の自動再起動）とセットで罠自体を消せる。
- **難易度**: 小 / **優先度**: P1

### S5. ブラウザ操作タイムアウトの設定一元化

- **現状の課題（根拠）**: Playwright のタイムアウトが各サービスにハードコードされ分散している。
  例: `next_engine_downloader.py` の `page.goto(LOGIN_URL, ..., timeout=60000)` と
  `expect_download(timeout=90000)`、`next_engine_invoice.py` の `expect_download(timeout=180000)`。
  回線やサイトが遅い日には一律に時間切れで失敗し、コードを直して再起動しないと回避できない。
- **改善案**: 設定モジュール（例: `portal_app/settings.py` を新設、または `portal_app/env.py` に
  `env_int()` ヘルパを追加）で `KURIMA_NAV_TIMEOUT_MS`（既定 60000）/
  `KURIMA_DOWNLOAD_TIMEOUT_MS`（既定 180000）を定義し、まずログイン・`goto`・
  `expect_download` の 3 系統から順次置換する。既定値は現行値のままにして挙動を変えない。
- **期待効果**: 遅延日でも `.env` 1 行の変更（＋再起動）で回避でき、業務が止まらない。
- **難易度**: 中 / **優先度**: P2

### S6. logs/ の保持期間管理と自動クリーンアップ

- **現状の課題（根拠）**: リポジトリ内 `logs/` は実測 28MB。`logs/jobs/history.jsonl` は
  追記のみで無限成長し、`read_job_history()` は毎回全行を読む（`progress_jobs.py`）。
  `/logs` 画面も毎回 `LOGS_ROOT.rglob("*")` で全走査する（`main.py` の `logs_index()`）ため、
  ジョブ詳細ログ（`logs/execution_runs/` 配下）が溜まるほど一覧・履歴が遅くなる。
  トップ層には `uvicorn-8002.out.log` 等の開発残骸も散在している。
- **改善案**: 保持日数 `KURIMA_LOG_RETENTION_DAYS`（既定 30）を新設し、サーバー起動時に
  `logs/execution_runs/` 配下で期限超過した run フォルダを削除する処理を `main.py` 起動時
  （または `progress_jobs.py` のジョブ終了時）に追加。`history.jsonl` は一定行数
  （例 2000 行）を超えたら末尾のみ残すコンパクションを行う。
- **期待効果**: `/jobs`・`/logs` の表示速度を長期運用でも維持。ディスクと同期の無駄を削減。
- **難易度**: 小〜中 / **優先度**: P2

### S7. 変換ロジックの単体テスト拡充と配布前チェック

- **現状の課題（根拠）**: `tests/` にあるのは `test_shipment_confirmation.py` 1 本のみ。
  `pyproject.toml` に pytest の依存宣言がなく、CI 定義（`.github/`）も無い。一方で
  住所補正（`yamato_conversion.py`。「様方/方」「A1/A-1 形式」等の複雑な規則を持つ）や
  クリックポストCSV生成のような純ロジックが多く、劣化しても実行するまで気づけない。
  配布は `update.bat`（git pull）で全PCへ即波及するため、壊れた版が届くとその日の業務が止まる。
- **改善案**: `pyproject.toml` に dev 依存として pytest を宣言し、`yamato_conversion` の
  住所補正・`paths.latest_order_csv` の選択規則・クリックポストCSV整形など純関数のテストを
  `tests/` に追加。GitHub Actions（`.github/workflows/test.yml`、windows-latest + uv）で
  push ごとに `uv run pytest` を回し、「テストが通った main だけを update.bat が拾う」状態にする。
- **期待効果**: 配布前に回帰を検知でき、「更新したら壊れた」を未然に防ぐ。
- **難易度**: 中 / **優先度**: P2

## カテゴリ別詳細 — 使いやすさ（7 件）

### U1. エラーメッセージの日本語対処ガイド化

- **現状の課題（根拠）**: ジョブ失敗時、`progress_jobs.py` の `fail()` は
  `message="処理中にエラーが発生しました。"` と `error=str(exc)` を保存し、画面は
  `errorBox.textContent = job.error`（`yamato.html`）で**生の例外文字列**を表示する。
  Playwright の時間切れは `Timeout 60000ms exceeded.` のような英語のまま出るため、
  非エンジニアは次の一手が分からない。一方 `yamato_b2_import.py` には既に
  `B2_LOGIN_STATES`（`still_on_login` / `needs_2fa` / `system_error` / `time_outside`）という
  原因分類と日本語文言があり、部分的には解決済み。
- **改善案**: `portal_app/services/error_hints.py` を新設し、例外型・キーワードから
  「原因と対処」を引く辞書を実装する（例: `TimeoutError` →「サイトの応答待ちで時間切れです。
  再実行してください。続く場合は回線を確認」/ ポータル未検出の `FileNotFoundError` →
  「`KURIMA_PORTAL_ROOT` の設定と SharePoint 同期を確認」/ `B2LoginError` → state 別の既存文言）。
  `ProgressJob` に `hint` フィールドを追加して `snapshot()` 経由で返し、各テンプレートの
  errorBox の下に対処文と「詳細ログを見る」リンク（`/logs/view?path=...`）を表示する。
- **期待効果**: 失敗時に自力復旧できる率が上がり、業務の中断時間と問い合わせが減る。
- **難易度**: 中 / **優先度**: P1

### U2. /logs 画面から共有ログ（SharePoint側）も閲覧可能に

- **現状の課題（根拠）**: `/logs` は `LOGS_ROOT`（リポジトリ内 `logs/`）だけを一覧する
  （`main.py` の `logs_index()`）。しかし実行ログ・エラーログの実体は
  `RUNTIME_LOG_DIR`（SharePoint の `神里\くりまポータルエラーログ`、`log_paths.py` で解決）に
  あり、画面から見えない。「/logs を開いたのに portal-error.log が無い」という迷子が起こる。
- **改善案**: `logs_index()` の一覧に `RUNTIME_LOG_DIR` を第 2 のソースとして追加し、
  「共有」ラベル付きで並べる。`_resolve_log_path()` の許可ルートを
  `LOGS_ROOT` と `RUNTIME_LOG_DIR` の 2 つに拡張する（絶対パス拒否・`resolve()` 後の
  ルート内検証という現行のパストラバーサル対策はそのまま維持）。
- **期待効果**: エラー調査がブラウザ内で完結し、エクスプローラで SharePoint を掘る作業が不要になる。
- **難易度**: 小〜中 / **優先度**: P1

### U3. トップページにデータ鮮度の警告と直近ジョブ表示

- **現状の課題（根拠）**: `main.py` の `dashboard()` は最新受注明細CSVのファイル名と
  更新日時（freshness）を表示するだけで、**どれだけ古くても見た目が変わらない**。
  また前夜のジョブが失敗していても、`/jobs` を自分で開かない限り気づけない。
- **改善案**: 警告しきい値 `KURIMA_FRESHNESS_WARN_HOURS`（既定 24）を新設し、超過時は
  `dashboard.html` で鮮度カードを警告色にして「在庫明細確認から再取得」への導線を出す。
  あわせて `read_job_history(limit=3)` の結果を「直近の実行」カードとしてトップに表示し、
  失敗（`failed`）は赤で目立たせる。
- **期待効果**: 朝イチの画面だけで「データが古い」「昨夜失敗していた」に気づける。
- **難易度**: 小 / **優先度**: P2

### U4. 実行中ジョブへの再アタッチとナビバッジ

- **現状の課題（根拠）**: 進捗表示はページ内 JS の `poll()` 頼みで、リロードやページ移動で
  `currentJobId` が消えると、ジョブは裏で動き続けているのに**画面からは見えなくなる**
  （`/jobs` の履歴に載るのは終了後のみ）。サーバー再起動後は `/progress/{job_id}` が
  404 になる（`progress_jobs` はメモリ保持、`main.py` の `progress_status()`）。
  「動いているのか分からず、もう一度押す」という S3 の温床にもなっている。
- **改善案**: (1) `GET /progress/running` を新設（`ProgressJobStore` に `list_active()` を追加）、
  (2) `base.html` のナビ「実行履歴」に実行中件数バッジを表示、(3) 各画面のロード時に
  running 一覧を取得し、同じ workflow のジョブがあれば進捗パネルへ自動で再アタッチする。
- **期待効果**: 「消えたように見える」不安と二度押しをなくし、LAN の別PCからも実行中が見える。
- **難易度**: 中 / **優先度**: P2

### U5. 実行履歴 /jobs のフィルタ・絞り込み

- **現状の課題（根拠）**: `/jobs` は「新しい順・最新200件」固定で（`jobs.html`、
  `main.py` は `read_job_history(limit=200)`）、結果・種別での絞り込みができない。
  「先週のクリックポストの失敗だけ見たい」時に目視で探すことになる。
- **改善案**: `jobs_history()` にクエリパラメータ `status` / `workflow` / `days` を追加して
  サーバー側でフィルタし、`jobs.html` に「結果」「種別」のセレクトと「失敗のみ」ショートカット
  ボタンを置く。実装は読み出し後の list 内包表記＋テンプレートの select 追加のみ。
- **期待効果**: 障害調査・振り返りの到達時間を短縮。
- **難易度**: 小 / **優先度**: P3

### U6. 設定画面 /settings の新設（.env の見える化）

- **現状の課題（根拠）**: 設定変更の手段は「`notepad .env` で編集して再起動」だけ
  （README セットアップ節）。どのキーが設定済みかは `scripts/doctor.py` を
  コマンドラインで実行しないと分からず、非エンジニアにはキー名の typo も検知できない。
- **改善案**: 段階 1: 読み取り専用の `GET /settings` を追加し、doctor 相当の情報を画面化する
  （認証系キーの設定状況を値マスクで表示・パス解決結果・稼働バージョン〔S4〕・ログ出力先）。
  段階 2: 秘匿でない動作設定（`KURIMA_FRESHNESS_WARN_HOURS` や S5 のタイムアウト等）に限り
  フォーム編集 → `.env` へ書き戻しを実装。反映に再起動が必要なキーには「要再起動」ラベルと
  `scripts/restart.bat` の案内を添える。
- **期待効果**: 設定状態の見える化で「動かない原因が設定だった」の調査が速くなる。
- **難易度**: 中〜大 / **優先度**: P2

### U7. ジョブ完了のブラウザ通知（進捗JSの共通化とセット）

- **現状の課題（根拠）**: NE取得やB2取込など数分かかるジョブは、完了までタブを見張る必要がある。
  また進捗ポーリング `poll()` の実装が `yamato.html` / `inventory.html` / `clickpost.html` /
  `shipment_confirmation.html` の 4 か所にコピーされており、進捗UIを直すたびに 4 ファイル
  修正が必要（挙動が微妙にズレる温床でもある）。
- **改善案**: 進捗JSを `portal_app/static/progress.js` に共通化した上で、完了/失敗時に
  `document.title` を「〔完了〕くりまポータル」等に変更し、Notification API による
  デスクトップ通知を追加する（許可はジョブ初回開始時に要求。拒否時はタイトル変更のみ）。
- **期待効果**: 見張り時間をゼロにし、他作業をしながら完了に気づける。進捗UIの保守も 1 か所で済む。
- **難易度**: 小〜中 / **優先度**: P3

## カテゴリ別詳細 — 運用・配布・安全（4 件）

### O1. update.bat に「更新→再起動」の一括オプション

- **現状の課題（根拠）**: `update.bat` は git pull → uv sync → Playwright 確認まで自動だが、
  最後は「サーバー起動中の場合は scripts\restart.bat で再起動してください」の**表示のみ**。
  再起動を忘れると旧コードが動き続ける（uvicorn は自動反映しない。S4 と同根で、
  README トラブルシュートにも記載のある既知の罠）。
- **改善案**: `update.bat` の末尾に `choice /c YN /t 30 /d N /m "サーバーを再起動しますか"` を
  追加し、Y なら `call scripts\restart.bat` を実行する（無応答 30 秒で N＝現状どおり案内のみ、
  無人実行でも安全）。ポートは restart.bat 側の既定（`KURIMA_PORT` → 8006）に任せる。
- **期待効果**: 「更新したのに直らない」を手順レベルで根絶。更新作業がダブルクリック 1 回で完結。
- **難易度**: 小 / **優先度**: P1

### O2. ホストPCの自動起動・自動復旧（常駐化）

- **現状の課題（根拠）**: 本番はホストPC 1台で `serve.ps1 -Mode lan` を手動起動する運用
  （README「LAN 共有」節）。Windows Update 等でホストPCが再起動すると、誰かが手で
  起動し直すまで全PCで使えない。`scripts/` に自動起動・監視の仕組みは存在しない
  （serve.ps1 / restart.bat / doctor.py 等のみ）。
- **改善案**: `scripts/register-autostart.ps1` を新設し、`schtasks` でログオン時に
  `serve.ps1 -Mode lan` を起動するタスクを登録する。あわせて 5 分毎に
  `http://127.0.0.1:8006/health`（`main.py` の `/health`）を叩き、無応答なら
  `restart.bat` を呼ぶ watchdog タスクも同スクリプトから任意で登録できるようにする。
- **期待効果**: 「朝つながらない」「PC再起動後に立ち上げ忘れ」をなくし、常駐を前提にできる。
- **難易度**: 小〜中 / **優先度**: P2

### O3. LAN公開時の簡易アクセス制御

- **現状の課題（根拠）**: LAN モードは `0.0.0.0` で待ち受け（`serve.ps1`）、アプリに認証は
  一切ない。クリックポストの**実決済**・ヤマトB2の**実取込**ボタンが LAN 上の全端末に
  露出しており、README の「社内LAN限定」という運用上の注意だけが防壁になっている。
- **改善案**: `main.py` に軽量なミドルウェアを追加し、`KURIMA_ALLOWED_HOSTS`
  （カンマ区切りの IP / プレフィックス。例 `192.168.1.`）に一致しない `request.client.host`
  からのアクセスを 403 にする。未設定時は現状どおり全許可（後方互換）。第 2 段階として、
  実行系 POST（決済・取込の `/start` 系）のみ `KURIMA_UI_PIN` の入力を要求する案もある。
- **期待効果**: 意図しない端末・誤操作からの不可逆操作を構造的に抑止できる。
- **難易度**: 中 / **優先度**: P2

### O4. 設定リファレンスの一元化（README と実装の乖離解消）

- **現状の課題（根拠）**: 実装に存在するのに README の環境変数表に載っていない設定キーがある。
  例: `KURIMA_B2_CHROME_PATH` / `KURIMA_B2_CHROME_PROFILE` / `KURIMA_B2_CHROME_PORT` /
  `KURIMA_B2_OPEN_URL`（`b2_chrome.py`）、`KURIMA_SHIPMENT_BUYER_LOOKBACK_DAYS` ほか
  出荷確定の遡り日数 4 キー（`shipment_confirmation.py`、既定 20/20/30/30 日）。
  「どんな設定で挙動を調整できるか」がコードを読まないと分からない。
- **改善案**: `scripts/doctor.py` に「認識する全設定キー・現在値（秘匿はマスク）・既定値」の
  一覧出力を追加する。S5 の設定モジュール（`portal_app/settings.py`）にキー定義を寄せれば
  一覧は自動生成でき、README の表は「主要キー + 全キーは doctor 出力参照」に整理して
  乖離が再発しない構造にする。
- **期待効果**: 新PCセットアップや挙動調整時に「使える設定」を確実に見つけられる。
- **難易度**: 小 / **優先度**: P3
