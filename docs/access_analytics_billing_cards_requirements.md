# アクセス解析取得・請求関連取得カード 要件定義書

## 文書情報

- 対象: kurima-portal（`portal_tool`）業務ポータルへの新規カード追加
- 種別: 要件定義書（実装準備。本ドキュメント自体はコード変更を含まない）
- 作成日: 2026-07-12
- スコープ: 「アクセス解析取得」カードと「請求関連取得」カードの2つ

## 背景・目的

社内では Obsidian vault（`C:/Users/hppym/dev/obsidian-vault`）側で、楽天市場・Yahoo!ショッピングの
アクセス解析データおよび精算・請求データを Playwright で自動取得する手順・実装仕様が既に検証済みである。
具体的には次の6ノートに、対象URL・DOM識別子・CSVスキーマ・確定状態判定・機密性の扱いまで含めて
一次情報として記録されている。

- `40-dev-notes/分析アプリ/楽天市場-デバイス別アクセス数取得手順.md`
- `40-dev-notes/分析アプリ/Yahoo!ショッピング-デバイス別アクセス数取得手順.md`
- `40-dev-notes/分析アプリ/Yahoo!ショッピング-精算・請求・受取明細取得手順.md`
- `40-dev-notes/分析アプリ/Yahoo!ショッピング-精算請求受取-Playwright実装仕様.md`
- `40-dev-notes/分析アプリ/楽天市場-BillPay精算データ取得手順.md`
- `40-dev-notes/分析アプリ/楽天市場-BillPay精算データ-Playwright実装仕様.md`
- 索引: `分析アプリ.md`（アクセス数・広告・精算会計の分類）

これらはいずれも Obsidian 上の検証済みノートであり、実際に日次業務で使うには kurima-portal
（`portal_tool`）の業務ポータルからボタン操作で実行できる状態にする必要がある。本要件定義書は、
既存の「在庫明細確認」「ヤマト伝票」「クリックポスト」「出荷確定」と同じ並びに、新規カード
「アクセス解析取得」「請求関連取得」を追加するための要件を定義する。

## スコープ

### 対象

1. **アクセス解析取得カード**
   - 楽天市場：RMS商品ページ分析のデバイス別アクセス数CSV取得
   - Yahoo!ショッピング：商品分析CSV（商品別PV/UU）・全体分析CSV（デバイス別ストア全体PV/UU/セッション）取得
2. **請求関連取得カード**
   - Yahoo!ショッピング：精算明細・請求明細・受取明細の取得
   - 楽天市場：BillPay確定済み精算・事前確認の店舗別内訳書CSV（および表示情報CSV）取得

索引ノート（`分析アプリ.md`）記載の対象サイトは「楽天(MVP)・Yahoo・Shopify の3サイト」だが、
今回参照した6ノートは楽天市場・Yahoo!ショッピングのみを検証済みである。Shopify分は本要件の対象外。

### 対象外（明記）

- 上記2カードの実コード実装・テストコードの追加（本ターンはドキュメントのみ）
- 本番の認証情報（Next Engine/RMS/Yahoo!ビジネスID/BillPay等のID・パスワード）の投入・設定作業
- 既存カード（在庫明細確認・ヤマト伝票・クリックポスト・出荷確定）の変更
- 広告データ（楽天RPP広告、Yahoo!アイテムリーチ広告）の取得。索引ノートには広告カテゴリの
  ノートも存在するが、今回のカード追加スコープには含めない
- Shopify向けのアクセス解析・精算取得
- 分析用ダッシュボード（BIツール等での可視化）の構築。本要件は「データを安全に取得・保存する」
  までを対象とし、可視化・集計レポートは別スコープとする

## 利用者・ユースケース

- 想定利用者: 現在 kurima-portal を業務端末（LAN内、`KURIMA_ALLOWED_CLIENTS` の許可範囲）から
  使っている現場スタッフ・店舗運営担当者
- ユースケース1（アクセス解析取得）: 日次で「本日分の楽天・Yahoo!アクセス実績を取得したい」と
  思ったときに、ポータルのカードを開き、対象モール・対象日を指定してボタンを押すだけでCSVが
  取得・保存される
- ユースケース2（請求関連取得）: 月次の会計処理・広告費照合のタイミングで「対象月の精算・請求・
  受取明細（Yahoo!）」「対象精算回の店舗別内訳書（楽天BillPay）」を取得したい担当者が、対象月・
  帳票種別を指定して取得する。会計正本として使う場合は「確定済みのみ」を選べる

いずれも「初回ログイン・追加認証・二段階認証は人が行い、その後の画面遷移・CSV取得だけを
自動化する」という既存カード（ヤマトB2等）と同じ運用モデルを踏襲する。

## 機能要件

### カード1: アクセス解析取得

| 項目 | 内容 |
|---|---|
| 対象モール | 楽天市場、Yahoo!ショッピング |
| 楽天市場の取得帳票 | RMS「データ分析 > 3 アクセス・流入分析 > 商品ページ」の商品ページ分析CSV。デバイス区分 `PC` / `楽天市場アプリ` / `スマートフォン` の3種を毎回取得（合計検証が必要な場合のみ `すべて` も取得）。URLは `https://datatool.rms.rakuten.co.jp/access/item` |
| Yahoo!の取得帳票 | (a) 商品分析CSV（`販売管理 > 商品分析`、商品別PV/UU、デバイス区分なし=`unspecified`）。(b) 全体分析CSV（`販売管理 > 全体分析`、`PC`/`スマホWeb`/`アプリ`/`合算値` の4種、ストア全体のPV/UU/セッション） |
| 取得粒度 | 日次（1日指定）。期間を範囲指定するとCSVが期間合算になるため、時系列データとしては使わない。前日1日分を基本とし、初回は日付×デバイスでバックフィルする |
| 出力形式 | 元CSVをそのまま保存する raw（楽天=UTF-8 BOM付き、Yahoo=CP932）と、分析アプリ向けに列を正規化した staging（UTF-8 BOM付き）の2層 |
| 実行方式 | 既存カードと同様、ボタン押下で非同期ジョブとして実行し、進捗パネルで工程（ログイン確認／取得／検証・保存）を表示する |
| 想定所要時間 | 楽天3端末＋Yahoo!2帳票で数分程度（既存ヤマト伝票カードの複数ステップ実行と同程度を想定） |

### カード2: 請求関連取得

| 項目 | 内容 |
|---|---|
| 対象モール | Yahoo!ショッピング、楽天市場（BillPay精算） |
| Yahoo!の取得帳票 | 精算明細（`offset_YYYYMM.csv`、精算締め日ごと）、請求明細（`billing_YYYYMM.csv`、月単位）、受取明細（`receipt_YYYYMM.csv`、月単位）の3種。画面上の正式名称は「精算明細」（清算ではない） |
| 楽天BillPayの取得帳票 | 確定済み精算（`/settlement_result`）の店舗別内訳書CSV（`document-type=34`）と、事前確認（`/billing_check`）の店舗別内訳書CSV（`document-type=33`）。加えて確定画面の「表示情報のCSVダウンロード」（12列、18か月分の精算回一覧） |
| 確定状態の扱い | Yahoo!・楽天BillPayともに「未確定（provisional）／確定（final）／不明（unknown）／データなし（no_data）」の4状態を判定できることを必須要件とする（**2026-07-13 更新**: 当初は3状態としていたが、「対象月に明細が存在しない」＝`no_data` と「判定に失敗した」＝`unknown` は運用上まったく別物のため、実装に合わせて `no_data` を追加した）。会計正本として使う取得は「確定済みのみ」を選べること |
| 取得粒度 | Yahoo!は対象月（年月）指定。楽天BillPayは18か月表示から精算回を全件列挙し、対象精算回（最新1件／指定発行日／全件）を選べること |
| 出力形式 | raw（Yahoo!=CP932、楽天BillPay=CP932）をそのまま保存し、正規化スキーマ（long形式含む）へ変換する staging を分離する |
| 実行方式 | 既存カードと同様、非同期ジョブ＋進捗パネル。対象月・対象精算回・確定済みのみ等の条件はフォームで指定する |
| 印刷書類（任意） | Yahoo!の「印刷」はHTML印刷モードのPDF監査コピー（`generated_from_html_print_mode: true`, `yahoo_origin_pdf: false`）。必須要件ではなく、取得できると望ましい付加機能として扱う |

## 非機能要件（機密情報の取り扱い）

Obsidianノートは次の機密区分を明示している。

- 楽天市場アクセス解析: `sensitivity: internal`（店舗運営上の非公開情報）
- Yahoo!アクセス解析: `sensitivity: internal`（商品別アクセス・売上等の非公開情報）
- Yahoo!精算・請求・受取明細: 「財務高機密」（注文ID、売上、手数料、広告費、税額を含む）
- 楽天BillPay精算データ: `sensitivity: confidential`（社名、企業ID、店舗別ID、店舗URL、請求額、支払額、精算書番号を含む）

これを踏まえ、両カードとも次を必須要件とする。

1. 生データ（raw CSV／PDF／ZIP）、認証情報、Cookie、`storageState`、ブラウザプロファイルは
   Obsidian vault・Gitリポジトリへ一切保存しない。
2. 保存先は、既存 `portal_tool` の Git管理外・非同期のローカル領域（`data/` 配下。既存
   `.gitignore` で `data/` 全体・`*.csv`・`*.pdf` 等が除外済み）に限定する。SharePoint同期の
   ポータルルート（既存の受注明細CSV等が置かれる共有ライブラリ）には**置かない**。特に
   「請求関連取得」カードのデータは全スタッフが閲覧できるSharePoint共有と混在させない。
3. 実社名・実金額・実ID・実URL・注文ID・精算書番号などの実データを、通常の実行ログ
   （`portal-run.log` 等）・進捗パネルの表示テキスト・監査JSONLへ出力しない。件数・状態・
   ハッシュ（SHA-256）・ファイル名など非機密メタデータのみを記録する。
4. **（2026-07-13 更新：自動ログインを許可）** 当初は「認証情報（ID・パスワード）は自動入力しない。
   初回ログイン・追加認証・二段階認証は人が行う（既存ヤマトB2カードと同じ境界）」としていたが、
   ユーザーの明示許可により、**本2カードの4フローに限り自動ログインを実装する**ことを要件とする
   （2026-07-13 に4サービス全てで自動ログイン＋CSV実取得の成功を確認済み）。ただし次を必須条件とする。
   - 認証情報は `.env`（Git管理外・SharePoint同期外）にのみ置く。環境変数
     （`KURIMA_RAKUTEN_KANRI_LOGIN_ID` / `KURIMA_RAKUTEN_KANRI_PASSWORD` /
     `KURIMA_RAKUTEN_LOGIN_ID` / `KURIMA_RAKUTEN_LOGIN_PASSWORD` /
     `KURIMA_YAHOO_LOGIN_ID` / `KURIMA_YAHOO_LOGIN_PASSWORD` /
     `KURIMA_BILLPAY_LOGIN_ID` / `KURIMA_BILLPAY_LOGIN_PASSWORD`）が
     **未設定の場合は自動ログインを行わず、従来どおり人手ログインへ縮退する**こと。
   - 二段階認証・追加認証・CAPTCHA が要求された場合は自動化せず、人手対応へフォールバックすること。
   - 認証情報を通常ログ・進捗パネル・監査JSONL・manifestへ出力しないこと（上記3と同じ）。
   - Cookie が有効な間はログインフォームに到達しないよう、専用永続Chromeプロファイルを併用すること。
5. 「請求関連取得」カードは既存カードより機密度が高いため、少なくとも実行結果の詳細表示・
   ダウンロードリンクの露出範囲について、既存の `KURIMA_ALLOWED_CLIENTS`（LAN接続元IP制御）が
   有効であることを前提運用とし、将来的な画面単位アクセス制御の要否は詳細設計以降で検討事項
   として引き継ぐ（本要件定義書では必須要件にはしない）。
