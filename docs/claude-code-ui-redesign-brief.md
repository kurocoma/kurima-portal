# Claude Code UI Redesign Brief

## 目的

くりまポータルツールのポータルトップと `/yamato` 画面を、ITリテラシーが高くないユーザーでも迷わず使える業務画面へ整理する。

重要な前提:

- 処理内容、実行順、Playwright自動操作、CSV変換ロジックは変えない。
- 通常ユーザーは「実行」ボタンを押すだけで、Next Engine取得、配送情報変換、Yamato B2取込まで完了できる状態を維持する。
- `/yamato` の実行は実取込モードが既定。B2ブラウザはヘッド付きで起動し、処理後も残す。
- 納品書PDFダウンロードはNext Engineの受注ステータスを動かす。復旧操作は通常操作から明確に分離する。

## 現在の画面構成

### 共通レイアウト

- `base.html`
  - 上部ヘッダー: `くりまポータルツール`
  - ナビ: `ポータル`, `在庫明細確認`, `ヤマト伝票`, `クリックポスト`
  - CSSは単一の `portal_app/static/styles.css`

### ポータルトップ `/`

表示内容:

- ページ見出し: `業務ポータル`
- 右上アクション: `在庫明細確認`
- 実行環境パネル
  - ポータルルート
  - 商品マスタ
  - 受注明細CSV
  - パス検出エラー時は候補パスを表示
- 機能カード
  - 在庫明細確認
  - ヤマト伝票
  - クリックポスト

課題:

- 「今日何をすればよいか」が分かりにくい。
- 実行環境の詳細パスが最初から目立ち、非技術ユーザーには情報量が多い。
- 各機能の状態、直近実行結果、注意有無が分からない。

### ヤマト伝票 `/yamato`

初期表示:

- 最新 `ne-yamato*.csv` を元に変換プレビューを表示する。
- エラーがあれば上部に `エラー` パネル。
- 成功メッセージがあれば `完了` パネル。

通常操作:

- `一括実行` パネル
- フォーム: `POST /yamato/run-selected`
- 表示ボタン: `実行`
- 隠し項目:
  - `headed=on`
  - `import_mode=execute`
  - `slow_mo_ms=150`
  - `preview_limit=30`
- 詳細設定の3チェックボックス、既定はすべてON:
  - `next_engine_workflow`: ネクストエンジン側の処理
  - `shipping_conversion_workflow`: 配送情報の変換
  - `yamato_import_workflow`: ヤマトインポート

保守操作:

- `納品書印刷待ちへ復旧`
- フォーム: `POST /yamato/restore-print-wait`
- 入力:
  - `order_nos`: 対象伝票番号
  - `slow_mo_ms`: 既定150
  - `restore_mode`: `check` / `execute`
- 納品書PDF検証やテスト後のステータス復旧用。通常の本番実行では触らない。

結果表示:

- `結果詳細` details内にまとめて表示。
- `prepare_result`
  - 購入者データ件数/保存ファイル
  - 商品情報データ件数/保存ファイル
  - 納品書PDF件数/保存ファイル/スキップ理由
  - 配送情報CSV
  - B2取込CSV
  - 監査ログ
  - 整合性警告
- `b2_import_result`
  - ステップ
  - CSV
  - 行数
  - 取込可能
  - ブラウザ実行
  - CSV選択
  - 実取込
  - 状態
  - 画面URL
  - 監査ログ
  - デバッグHTML
  - スクリーンショット
  - 警告
- `restore_result`
  - 対象件数
  - 実行有無
  - 失敗伝票
  - 監査ログ
  - 伝票ごとの復旧前/復旧後/結果
- `result` 変換概要
  - 変換元CSV
  - 商品マスタ
  - 入力行数
  - 出力行数
  - 重複除外
  - 住所自動補正
  - 住所要確認
  - 出力CSV
  - 監査ログ
  - 警告
  - 住所補正結果テーブル
  - 変換プレビューテーブル

課題:

- 業務上重要な結果と、開発者向けのパス/ログ/デバッグ情報が同じ密度で並んでいる。
- 「実行中に何が起きるか」「完了後に何を確認すればよいか」が画面から読み取りにくい。
- エラー文言が例外文字列のままで出るため、利用者が次の行動を判断しにくい。
- 復旧操作は危険度が高いが、通常導線と同じ画面密度で見える。
- テーブルが横長で、確認が必要な行と単なるプレビュー行の優先度差が弱い。

## 守るべきフォーム契約

### `/yamato/run-selected`

フォーム名は維持する:

- `next_engine_workflow`
- `shipping_conversion_workflow`
- `yamato_import_workflow`
- `headed`
- `import_mode`
- `slow_mo_ms`
- `preview_limit`

既定値は維持する:

- 3チェックボックスはすべてON。
- `headed=on`
- `import_mode=execute`
- `slow_mo_ms=150`
- `preview_limit=30`

UI上の原則:

- 通常ユーザーには「実行」だけを主ボタンとして見せる。
- 3チェックボックスは「詳細設定」内に隠す。
- 伝票番号入力は通常実行には不要。表示しない。

### `/yamato/restore-print-wait`

フォーム名は維持する:

- `order_nos`
- `headed`
- `slow_mo_ms`
- `restore_mode`

UI上の原則:

- 「保守・復旧」または「テスト後の復旧」として、通常操作から明確に分離する。
- 本番の日次実行では使わないことが分かる文言にする。
- `execute` は状態を変更するため、視覚的に注意を促す。

## デザイン方針

対象ユーザー:

- ExcelやWeb操作には慣れているが、CSV、監査ログ、dry-run、Playwrightなどの用語には詳しくない。
- 日次業務として「ヤマト伝票を作る」ことが目的。

画面の優先順位:

1. 今日やることが分かる。
2. 押すボタンが分かる。
3. 実行結果が成功/要確認/失敗で分かる。
4. 要確認の場合、どの行を直せばよいか分かる。
5. 開発者向けのパス、ログ、スクリーンショットは必要な時だけ見られる。

推奨構成:

- ポータルトップ
  - 主要業務カードを「ヤマト伝票」「在庫明細確認」「クリックポスト」に整理。
  - ヤマト伝票を現在の主機能として一番目立たせる。
  - 実行環境パスは「環境情報」detailsへ格納。
  - 各カードに「何が作られるか」「いつ使うか」を短く記載。

- ヤマト画面
  - 上部に「ヤマト伝票を作成」主パネル。
  - 大きい主ボタン: `ヤマト伝票を作成する` など、業務名を含める。
  - 小さな説明: `印刷待ちのヤマトB2v6対象を取得し、B2へ取り込みます。`
  - 詳細設定は折りたたみ。通常は閉じておく。
  - 実行後は上部に結果サマリー:
    - 成功: `完了しました`
    - 要確認: `確認が必要な項目があります`
    - 失敗: `処理が止まりました`
  - 結果サマリーには件数を優先表示:
    - NE対象件数
    - B2取込CSV件数
    - 住所自動補正件数
    - 要確認件数
    - B2取込結果
  - 住所補正結果は「要確認」優先。自動補正のみの場合は折りたたみ。
  - プレビューとログは `詳細情報` / `開発者向け情報` に格納。
  - 復旧操作は `保守・復旧` detailsに分離し、通常実行の下部へ置く。

文言ルール:

- `CSV` は必要な箇所だけ。主導線では `配送データ`, `B2取込データ` と表現する。
- `監査ログ`, `debug HTML`, `skipped_reason` などは通常表示しない。
- `dry-run` という言葉は出さない。この画面の既定は実取込。
- エラーは「何が起きたか」「次に何をするか」に分ける。

ビジュアルルール:

- 業務ツールなので、落ち着いた実務向けUIにする。
- 画面全体をカードだらけにしない。カードは機能入口、結果サマリー、注意事項などに限定。
- カードやパネルの角丸は8px以下。
- 1366x768でも主ボタンと状態が見える。
- 長いファイルパスは折りたたみ、または横にはみ出さず省略/折返しする。
- テーブルは横スクロール可。ただし重要列は左に寄せる。
- 外部CDNやフロントエンドビルドは追加しない。

## 受け入れ基準

- `/` の最初の画面で、主要機能とヤマト伝票への入口が分かる。
- `/yamato` の最初の画面で、押すべきボタンが1つに見える。
- 詳細設定を開かなくても、本番の一括処理が実行できる。
- 結果表示は、成功/要確認/失敗と件数が先に分かる。
- 開発者向け情報は折りたたまれている。
- `main.py` の処理契約を壊していない。
